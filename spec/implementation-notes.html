Melissa Map Implementation Notes

Design Decisions

  • V1 is static-first: no live geocoding or corpus-building calls during gameplay.
  • Primary mode is click-the-intersection; other modes are deferred.
  • CSS should be token-driven so a later aesthetic pass can reskin without rewriting game logic.
  • Map/data V1 uses a dependency-free coordinate-map adapter because the app scaffold and package file had not landed yet. The adapter API is intentionally small so a MapLibre implementation can satisfy the same contract later.
  • Question order is deterministic by seed and intersection id, which keeps QA repeatable while avoiding hardcoded array order as gameplay order.
  • Guess resolution uses haversine distance between lat/lon coordinates; screen position is only used to convert a click into a guess coordinate.
  • Scoring is intentionally coarse. A click is correct when the requested intersection is the nearest known intersection within the snap radius; correct guesses display at the snapped intersection, while misses name the nearest known intersection and receive 1-5 closeness points instead of 0-100 coordinate points.
  • Seed corpus lives in src/data/intersections.json; src/data/intersections.ts adapts it into the map/data Intersection contract.
  • Seed coordinates are playable approximate points, with uncertainty documented in src/data/seed-notes.md.
  • App shell uses React + Vite + TypeScript with small components and a tokenized CSS layer for the later aesthetic pass.
  • Game-loop state consumes the map/data workstream exports instead of duplicating question selection or scoring.
  • The placeholder SVG grid was replaced with dependency-free raster tiles from CARTO basemaps. The app renders no-label tiles while guessing and fades in label tiles after reveal.
  • The map uses actual CARTO Voyager no-label tiles while guessing. Each raster tile is post-processed in a canvas to quiet the yellow road palette while preserving real CARTO road geometry.
  • Base tile processing now uses a raster heuristic to darken only thicker yellow road bands, keeping smaller local streets quieter than the earlier broad yellow-road recolor.
  • Reveal mode processes CARTO label tiles separately: labels keep CARTO's native placement, but the raster text is darkened and given a white halo for readability. The adaptive white-text-on-dark-road experiment was rejected because it created artifacts on dark major-road labels.
  • The map default now focuses on the developed Las Vegas valley rather than the full Lake Mead / far desert extent, but starts slightly farther out than the first tuned pass so more valley context is visible.
  • Map controls support zoom in, zoom out, reset, drag-to-pan, and wheel zoom without adding MapLibre.
  • The starting valley frame is also the minimum zoom level for the current viewport. Wheel zoom uses normalized delta values batched through animation frames so mouse wheels and trackpads feel closer in speed.
  • After each map render, the adapter preloads a capped set of center-near CARTO base and label tiles for the next zoom level, including processed tile variants, to soften the first zoom-in.
  • Reveal mode preserves the current zoom and only nudges the center when the guess or answer would be too close to the viewport edge.
  • Starting a fresh unrevealed question restores the default valley frame so any reveal/inspection zoom does not carry forward.
  • Close marker labels split left/right to avoid stacking when the guess is nearly correct.
  • Reveal mode adds a high-contrast answer-street callout near the correct marker using the current intersection's primary and cross streets, so the answer remains readable even when raster tile labels are small.
  • Wrong reveals no longer render generated learning labels from the seed corpus. The improved CARTO label layer now carries nearby street learning context, while the answer callout remains the explicit exact-answer anchor.
  • Result feedback can call out when the guess appears close to one road but misses the cross street.
  • The Lake Mead Boulevard / Las Vegas Boulevard North seed point is explicitly guarded in validation after QA found it had drifted west to the I-15/Lake Mead interchange.
  • Area-specific play uses a normalized area bucket separate from the descriptive region field. This keeps the UI stable while preserving more granular seed notes.
  • The selected area is stored with local progress. Changing areas resets the current deck for that area; using Reset clears progress while preserving the selected area.
  • Coordinate QA now has two layers: dependency-free static guards in npm run validate:seed for known corrected rows, plus network-backed OSM road-crossing verification in npm run verify:seed:coords.
  • The Spring Mountain / Paradise prompt was corrected to Sands Avenue / Paradise Road because OSM road geometry shows the east-of-Strip crossing under the Sands Avenue name.
  • The invalid/non-crossing Flamingo / Lamb and Spring Mountain / Maryland rows were replaced with OSM-verifiable Flamingo / Pecos and Desert Inn / Maryland rows.

Deviations

  • MapLibre is still not installed. The current V1 map is a custom lightweight tile adapter using Web Mercator tile math, which keeps the app dependency-light while showing a real Vegas basemap.
  • The road-read feedback is heuristic. It infers street orientation from the seed corpus and should be reviewed as more intersections are added.

Tradeoffs

  • Manual seed corpus is faster and safer for V1 than building the full OSM/Clark County ETL immediately.
  • MapLibre is preferred, but a coordinate-map fallback is acceptable if it keeps V1 playable quickly.
  • The sample corpus remains available for tiny fixtures, but app gameplay should use the 76-item intersections export.
  • The first corpus started at 70 intersections and was expanded to 76 to give every exposed area bucket at least 10 playable prompts.
  • The basemap now depends on public CARTO tile availability. A future production pass should decide whether to self-host tiles, use PMTiles, or move to MapLibre with a managed style.
  • Zoom/pan is intentionally modest for V1. It is enough for reading the local road grid, not a full slippy-map replacement.
  • Tile preloading is capped per layer and only looks one tile-zoom level ahead. This avoids large background fetches while improving the jarring first zoom-in case.
  • Reveal no longer zooms in to expose labels because that removed too much location context; split labels and a small preserve-zoom pan are the current compromise.
  • The answer-street callout duplicates information that may also exist in tile labels, but it keeps the reveal legible without forcing zoom or depending on third-party label density.
  • The processed CARTO labels are more geographically faithful than generated DOM labels, but the halo is still raster-based rather than true vector text styling.
  • Tile post-processing depends on CARTO returning CORS-enabled PNGs. If base or label processing fails, the image falls back to the original CARTO tile instead of blanking the map.
  • Spot-check coordinate guards are intentionally narrow for now; broader corpus precision still needs a future ETL or review pass.
  • The static build is deployed to the permanent here.now site at https://clever-grove-3t69.here.now/. The live app was browser-verified after the 1-5 closeness scoring change.
  • Playable area buckets require at least 10 intersections before being shown. Henderson/Green Valley and North Las Vegas/Northwest are being topped up manually rather than splitting them into smaller sparse regions.
  • The first area chooser is a native select in the sidebar instead of a full start screen; this preserves the fast first prompt while still letting the player choose All Areas or a specific bucket before guessing.
  • The OSM verifier is intentionally separate from the normal build because it depends on public Overpass availability and live map data.
  • Rows that do not produce an OSM road crossing should be treated as content QA failures, not just coordinate misses; the verifier exposes those instead of hiding them behind approximate points.
  • The here.now site was redeployed after the coordinate QA fixes; refreshing the live page should load the bundle with the corrected Lake Mead Parkway / Boulder Highway point.

Open Questions

  • Future QA should tune the snap radius as the corpus grows denser, especially in areas with closely spaced intersections.
  • Integration follow-up: replace the coordinate-map adapter with MapLibre behind the same adapter contract when the aesthetic/map pass is ready.
